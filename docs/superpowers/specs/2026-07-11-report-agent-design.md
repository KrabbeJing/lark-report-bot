# 数金小助手日报、周报与月报智能体设计

## 文档状态

本文档记录 2026-07-11 确认的完整业务方案，并覆盖旧设计中与以下内容相关的规则：

- 日报多来源冲突处理。
- 多日日报进入 AI 汇总时的去重方式。
- 周报模板创建、AI 预填、人工修改、海报与群聊推送。
- 月报生成与多人协作。

`2026-07-03-daily-fact-data-layer-design.md` 仍用于说明第一期事实层背景和已有实现；如规则不一致，以本文档为准。

## 目标

在保留群聊日报和表单日报两种提交习惯的前提下，形成一条可追溯、可人工干预、可定时运行的团队报告链路：

```text
群聊日报 -> 群聊日报原始表 ----\
                                  -> 日报统一事实表 -> AI 周报草稿 -> Wiki 周报 -> 海报/群聊
表单日报 -> 现有表单日报表 ------/                         |
                                                           -> 月报草稿 -> 多人协作
```

系统的智能性主要体现在事实约束下的归纳、分群、风格适配和异常处理，不以 AI 替代确定性去重、指标填写或人工最终口径。

## 技术方案

- 现有 Node.js 服务继续作为业务核心。
- 腾讯云和 PM2 继续承载常驻进程、飞书长连接和定时任务。
- 飞书多维表格作为事实层、配置中心、实例状态和审计记录载体。
- 飞书 Wiki 电子表格作为正式周报协作界面。
- 现有 Node.js 海报模块继续从最终 Wiki 表格生成图片。
- 外部国产模型通过 OpenAI 兼容适配层接入，具体厂商在联调阶段选择。
- 当前不建设妙搭后管。团队规模不超过 50 人时，直接使用多维表格视图和权限协作。

## 日报事实层

### 事实粒度

`日报统一事实表`坚持一人一天一条有效事实记录，唯一键为：

```text
优先：成员OpenID + 日报日期
兜底：日报提交人姓名 + 日报日期
```

日报归属由`日报日期`决定，不由提交时间决定。

### 来源时间

- 群聊日报：使用飞书消息发送时间。
- 表单日报：优先使用记录最后修改时间，缺失时使用记录创建时间。
- 两个来源时间完全相同时，表单优先。

事实表新增或明确维护：

- `来源时间`
- `有效来源`
- `自动处理说明`
- `分管领导`

### 多来源冲突状态

| 场景 | 最终内容 | 合并状态 | 冲突状态 | 事实记录状态 |
| --- | --- | --- | --- | --- |
| 只有一个来源 | 直接采用 | `单来源` | `无冲突` | `有效` |
| 群聊与表单内容一致 | 合并来源引用 | `重复已合并` | `无冲突` | `有效` |
| 群聊与表单内容不一致 | 按来源时间取最新 | `按时间取最新` | `已自动处理` | `有效` |
| 无法匹配成员 | 保留记录，暂不进入正式汇总 | `单来源` | `无冲突` | `待人工确认` |

`已自动处理`表示出现过真实差异，但系统已经按确定规则解决。`自动处理说明`记录采用来源和原因，例如“表单最后修改时间较晚，采用表单内容”。

第一期不使用 AI 合并冲突内容，也不要求提交人选择。后续复杂场景再增加 AI 合并或人工选择机制。

`事实记录状态`固定使用三个选项：

- `有效`：可以进入周报、小群总结和月报的数据范围。
- `待人工确认`：数据保留但默认不进入正式汇总，修正人员匹配或异常后再转为`有效`。
- `忽略`：明确排除的数据，永久不进入 AI 汇总，但原始记录和来源引用继续保留。

### 群聊多版本

- 原始表保留同一成员同一天的所有群聊日报。
- 最新消息为`主版本`，旧消息为`历史版本`。
- 事实表只采用最新主版本参与来源比较。
- 迟到日报不发送通知。

### 多日日报

多日日报继续按日期拆成多条事实记录，以保证完整率、日期筛选、跨周归属和唯一键稳定。

AI 汇总前按一次原始提交去重：

- 同一`来源消息ID`或`来源记录ID`拆出的多日事实，只作为一条 AI 输入。
- AI 输入携带本周内的日期覆盖范围，正文只出现一次。
- 不同来源 ID 即使文字相同也不在数据层删除，可由 AI 归纳为连续推进事项。

## 组织与配置

### 团队通讯录

团队通讯录至少维护：

- 团队成员和成员真实姓名。
- 当前 OpenID、历史 OpenID、成员别名。
- 所属科室或板块。
- 敏捷小组。
- 直属上级。
- 分管领导。
- 成员状态。

三个分管领导群通过`分管领导OpenID + 群聊ChatID`映射。每个群只接收对应分管范围的数据。

`直属上级`和`分管领导`必须独立维护：直属上级表示组织架构中的直接汇报关系；分管领导表示成员当前工作的业务板块归属。成员 A 的直属上级可以是 B，但其日报事实和周报仍按分管领导 C 进入 C 管理的板块及群聊。

### 多维表格配置中心

建议逐步增加：

- `周报配置表`
- `指标负责人配置表`
- `敏捷小组配置表`
- `分管领导与群聊配置表`
- `周报实例表`
- `AI生成记录表`
- `AI风格样例表`
- `月报配置表`
- `月报实例表`
- `任务执行记录表`

Token、密钥和飞书应用凭证不写入 Base，继续使用服务端环境变量或受控配置文件。

## 周报模板

### 每周实例

- 每周一 09:00 自动复制固定模板，生成本周工作表副本。
- 周报实例唯一键为`ISO年份 + ISO周次`，重复执行不得创建第二份。
- 自动填写周报名称和报告周期。
- 在`周报实例表`记录 spreadsheet token、sheet ID、周期和状态。
- 模板复制失败时重试并通知管理员。

### 动态定位

不再把`C26`等坐标作为业务配置，而是读取模板文本后按以下路径定位：

```text
模块标题 + 项目/指标/部门名称 + 内容类型
```

合并单元格扫描时继承其左上角或上方业务标题，避免空白行导致上下文丢失。

#### 模块一：核心指标完成情况

- 指标值由负责人手工填写，AI不得推算或补写。
- 每个指标在配置表中绑定一名负责人和目标定位信息。
- 未填写不阻塞周报和海报生成。

#### 模块二：敏捷项目组工作进展

每个敏捷组包含两个独立目标 Cell：

- `本周重点事项说明`
- `下周工作计划`

每个 Cell 可以包含多条、换行和编号，不设置业务条数或字符数上限，不向相邻单元格溢出，也不机械截断。

日报中的问题只有在仍未解决且会影响项目进展时，才基于明确事实写入`本周重点事项说明`。

#### 模块三：部门管理工作

定位规则为：

```text
模块 + 部门管理板块 + 内容类型 -> 右侧连续三个目标 Cell
```

- 每个 Cell 写一条。
- 可以写 0 至 3 条，未使用 Cell 保持空白。
- 模型输出和 Node.js 写入均限制最多 3 条，绝不写入下一板块。
- 每个板块独立配置来源科室、敏捷小组、指定人员和负责人。

## AI 生成规则

### 事实边界

所有周报、小群总结和月报统一遵守“无依据不生成”：

- 不改变业务名称、指标名称和既有口径。
- 不修改来源中的数字、比例、日期、状态和责任人。
- 不把“推进中、计划中、待确认”改写为“已完成”。
- 只能归纳、合并重复事项、调整顺序和压缩表达。
- 下周计划只提取日报中明确出现的计划；无明确计划时保持空白。
- 核心指标完全使用负责人填写值。
- 确定性规则无法解决的异常内容不生成，并进入异常记录。

### 风格适配

风格样例粒度为：

```text
模块 + 具体敏捷组/部门管理板块 + 内容类型
```

- 每类建议提供 3 至 5 条人工确认后的历史正式周报。
- 历史样例只决定表达方式，当前日报只决定事实。
- 不得把样例中的旧项目、数字和状态带入新周报。
- 无板块专属样例时回退到模块通用风格。
- 优先使用最新且状态为`已确认`的样例。

### 生成审计

`AI生成记录表`至少保存：

- 周期和生成批次。
- 目标模块、板块、内容类型和目标 Cell。
- AI 输出内容及内容哈希。
- 来源事实记录 ID。
- 使用的模型、提示词版本和风格模板版本。
- 生成时间、写入时间和写入状态。

Wiki 展示干净的业务内容；审计表用于追溯，不在普通运行日志中重复记录完整日报。

### 人工修改保护

- AI 首次写入后保存每个 Cell 的内容快照和哈希。
- 刷新时，当前内容与上次 AI 内容一致才允许覆盖。
- 当前内容不同则视为人工修改，不再覆盖。
- AI 写入后被人工清空也属于人工修改，保持空白。
- 模块三三个 Cell 分别判断，不因一个 Cell 修改而锁定整个区域。

## 周报时间线

周报周期为周一至周五，按`日报日期`归属：

| 时间 | 动作 |
| --- | --- |
| 周一 09:00 | 复制模板并登记本周周报实例 |
| 周五 17:00 | 检查核心指标，仅提醒未填写负责人一次 |
| 周五 18:00 | AI 预填 Wiki 周报，标记“AI总结生成，仅供参考” |
| 周六 09:30 | 纳入新日报，刷新仍由 AI 管理的 Cell |
| 周六 10:30 | 本周数据截止 |
| 周六 10:50 | 读取最终 Wiki 内容并生成海报 |
| 周六 11:00 | 向部门大群发送海报 |
| 周六 11:05 | 向三个分管领导小群分别发送 AI 周报 |

- 周五晚至周六 10:30 补交的周五日报进入本周周报。
- 周六 10:30 后提交的日报正常入事实表，不自动更新或重发本周海报。
- 迟到日报不额外通知任何人。
- 所有时间通过配置修改，默认使用`Asia/Shanghai`。

## 分管领导小群

- 使用同一周报周期和周六 10:30 截止时间。
- 按分管领导筛选事实数据，三个群完全隔离。
- 周六 11:05 发送飞书消息卡片或富文本，不生成海报。
- 明确标记`AI总结生成，仅供参考`。
- 不等待负责人审核，不因个别成员缺日报而阻塞。

## 海报发布

- 海报只读取周六 10:30 后的最终 Wiki Cell 值，不直接读取 AI 原始输出。
- 海报文本不使用图片生成模型，继续由确定性的 Node.js 模板生成。
- 10:50 开始生成和校验，11:00 前最多重试 3 次。
- 校验报告周期、图片非空和尺寸合理。
- 禁止发送上周海报、空白海报或错误图片。
- 11:00 仍失败时，部门大群发送正式 Wiki 链接卡片，管理员收到失败原因。
- 修复后是否补发海报由管理员决定，系统不自动重复发送。

## 月报

### 数据源与归属

- 月报以状态为`已发布`的 Wiki 周报最终内容为主要事实来源。
- 跨月周在`周报实例表`配置`月报归属月份`，默认可按周五月份生成并允许人工调整。
- 同一周只归属一个月。
- 归属上月但尚未发布的周报标记`待周报补齐`，发布后只补充未被人工修改的月报章节。

### 协作流程

| 时间 | 动作 |
| --- | --- |
| 每月 1 日 09:00 | 复制月报模板、AI 预填、通知章节负责人 |
| 每月 2 日 17:00 | 仅提醒未完成章节负责人一次 |
| 每月 3 日 09:00 | 无论完成情况如何，通知统稿人当前状态 |

- 每个章节可以配置多名负责人。
- 整份月报配置一名统稿人。
- 状态流转：`AI草稿 -> 填写中 -> 章节已完成 -> 待统稿 -> 已确认`。
- 负责人直接在同一飞书文档协作，通过卡片按钮或 Base 标记章节完成。
- 首期不自动发送群聊，也不自动锁定文档。
- 3 日后发生周报补齐时，再通知统稿人一次。

## 模型与数据安全

- 模型层使用 OpenAI 兼容接口和可替换 Provider。
- 通过环境变量配置 Base URL、API Key、模型、超时和重试次数。
- 可以发送日报工作内容和成员姓名。
- 不发送 OpenID、群聊 ID、表格 Token 等系统标识。
- API Key 不写入 Base、配置样例或日志。
- 正式上线前确认服务商的数据留存和企业合规条款。

## 定时任务与幂等

- PM2 继续负责进程守护和自动重启。
- 调度器内存防重只作为单进程优化，不能作为业务幂等依据。
- 每个任务使用`业务周期 + 任务类型 + 目标对象`生成执行键。
- `任务执行记录表`保存计划时间、开始时间、完成时间、状态、重试次数和错误摘要。
- 周报实例、月报实例和消息发送都必须先查询持久化状态，防止 PM2 重启后重复创建或重复发送。

## 上线策略

1. 保持现有腾讯云、PM2、Gitee部署方式。
2. 先修复现有周报发送字段校验错误并完善安全日志。
3. 在个人组织建立新增表和字段。
4. 所有新任务默认关闭，提供手动执行和 dry-run。
5. 依次验证事实冲突、模板定位、AI 输出、人工修改保护、海报和群聊。
6. 手动完成一轮完整周报和月报演练。
7. 再逐项启用个人组织定时任务。
8. 正式组织重新创建应用、Base、Wiki、群映射和 OpenID 配置后重复验收。

## 分阶段交付

### 阶段一：事实层修订

- 按来源时间解决冲突。
- 更新单选值、来源字段和测试。
- 增加 AI 输入多日去重。

### 阶段二：周报实例与动态模板定位

- 自动复制模板和登记实例。
- 实现标题路径定位、合并单元格上下文和区域边界校验。
- 实现配置表读取与功能开关。

### 阶段三：AI 生成、风格与人工修改保护

- 强化结构化输出和事实校验。
- 增加风格样例、AI 生成审计和 Cell 快照。
- 实现周五预填和周六刷新。

### 阶段四：发布链路

- 指标提醒。
- 海报生成、校验、兜底和部门群发送。
- 三个分管领导小群汇总。

### 阶段五：月报

- 月报模板、章节配置、负责人协作和统稿状态。
- 跨月周归属与后补机制。

### 阶段六：正式组织迁移

- 替换租户资源 ID 和成员 OpenID。
- 最小权限授权、全链路演练和分阶段启用。

## 验收标准

- 群聊和表单日报均进入统一事实表，冲突按时间确定有效来源。
- 多日日报在事实层按天保存，在 AI 输入中按原始来源去重。
- 周报模板复制任务重复执行不会生成重复实例。
- 模板插入行后仍可按标题路径找到正确 Cell，不跨模块写入。
- 模块二单 Cell 内容不受三条限制；模块三最多写三个 Cell。
- AI 每条内容可追溯到事实记录，不改变数字、状态和业务口径。
- 人工修改或清空的 Cell 不被后续 AI 刷新覆盖。
- 指标空白不阻塞海报，且不会被 AI 补写。
- 迟到日报按截止规则处理，不通知、不自动重发。
- PM2 重启不会重复建周报、重复生成月报或重复发群。
- 海报失败时只发送正确 Wiki 链接，不发送错误图片。
- 月报支持多章节负责人和一名统稿人协作。
- 所有自动任务都可配置关闭，并能在个人组织完成 dry-run 验收。

## 实施前输入

以下信息不阻塞架构设计，在对应阶段开始前提供即可：

- 国产模型服务商、Base URL 和模型名称。
- 各敏捷组、部门管理模块的 3 至 5 条正式历史样例。
- 月报模板和章节负责人清单。
- 正式组织的应用、Base、Wiki、群聊和成员 OpenID 映射。
