# 数金小助手日报统一事实表数据层设计

> 状态说明：本文档记录第一期事实层的原始设计。2026-07-11 后确认的冲突状态、按来源时间取最新、多日数据进入 AI 前去重等规则，以 `2026-07-11-report-agent-design.md` 为准。

## 背景

数字金融部内有多个处室、板块和敏捷小组，不同团队提交日报的方式不同：

- 有的成员通过飞书多维表格关联表单提交日报，记录进入现有 `工作日报表/日报汇总表`。
- 有的成员习惯在项目组日报群直接发送日报消息。
- 部门存在分管领导管理多个处室/板块的情况，也存在敏捷小组成员跨处室/板块协作的情况。

现有机器人已支持从群聊识别日报并写入多维表格，也保留原有表格/wiki 链接生成海报能力。本设计聚焦第一期数据层闭环，为后续 AI 周报、领导审核、Wiki 周报和海报推送提供统一、可追溯的数据基础。

## 第一期开工范围

第一期只做数据层闭环：

- 群聊日报原始记录入表。
- 表单日报和群聊日报归一到统一事实表。
- 通讯录匹配成员、所属板块、敏捷小组、直属上级和分管领导。
- 同一成员同一天多来源日报去重、合并和冲突标记。
- 多日合并日报拆分为多条事实记录。

第一期暂不做：

- AI 周报摘要。
- 写入 Wiki 周报模板。
- 生成海报。
- 定时推送。
- 自动创建多维表格或字段。

## 总体方案

采用“事件实时 + 定时校准”方案。

```text
群聊日报消息 -> 群聊日报原始表 -> 日报统一事实表
表单日报记录 -> 工作日报表/日报汇总表 -> 日报统一事实表
```

在现有日报多维表格 Base 内手动新增两张表：

- `群聊日报原始表`：保存机器人从群聊抓到的每一条日报消息，作为群聊来源的原始凭证。
- `日报统一事实表`：统一承接表单日报和群聊日报的规范化结果，作为后续统计、AI 周报和领导审核的唯一分析入口。

现有 `工作日报表/日报汇总表` 不改字段、不影响当前表单和工作流。

## 数据表设计

### 群聊日报原始表

建议字段：

| 字段 | 类型建议 | 说明 |
| --- | --- | --- |
| 消息ID | 文本 | 飞书消息 ID，用于消息去重 |
| 群ID | 文本 | 飞书群 ID |
| 群名称 | 文本 | 可选，方便人工查看 |
| 发送人OpenID | 文本 | 消息发送人的 open_id |
| 标题姓名 | 文本 | 从日报标题解析出的姓名 |
| 日报日期范围 | 文本 | 例如 `2026-06-29~2026-06-30` |
| 拆分日期列表 | 文本 | 例如 `2026-06-29,2026-06-30` |
| 原始消息文本 | 长文本 | 群聊消息原文 |
| 解析后工作总结 | 长文本 | 去掉标题但保留编号的正文 |
| 内容指纹 | 文本 | 规范化正文后生成，用于重复和冲突判断 |
| 消息时间 | 日期时间/文本 | 飞书消息发送时间 |
| 接收时间 | 日期时间/文本 | 机器人接收并处理的时间 |
| 解析状态 | 单选/文本 | `已解析`、`低置信度`、`解析失败` |
| 原始记录状态 | 单选/文本 | `主版本`、`历史版本`、`解析失败` |

同一成员同一天在群里发多条日报时，原始表保留每一条。最后一条为 `主版本`，之前的记录标记为 `历史版本`。

### 日报统一事实表

建议字段：

| 字段 | 类型建议 | 说明 |
| --- | --- | --- |
| 事实唯一键 | 文本 | 优先 `open_id + 日报日期`，兜底 `姓名 + 日报日期` |
| 日报日期 | 日期 | 拆分后的单日日期 |
| 实际日报提交人 | 人员 | 当前飞书账号，可为空或显示外部账号名 |
| 日报提交人姓名 | 文本 | 业务展示姓名，周报和海报优先使用 |
| 成员OpenID | 文本 | 当前匹配到的 open_id |
| 所属板块 | 文本 | 从通讯录映射 |
| 敏捷小组 | 文本 | 从通讯录映射 |
| 直属上级 | 人员/文本 | 从通讯录映射 |
| 分管领导 | 人员/文本 | 从通讯录映射 |
| 今日工作总结 | 长文本 | 事实表主内容 |
| 明日工作计划 | 长文本 | 表单或群聊解析结果 |
| 遇到的问题 | 长文本 | 表单或群聊解析结果 |
| 内容指纹 | 文本 | 事实表主内容规范化后的指纹 |
| 日报来源 | 单选/文本 | `form`、`chat`、`form+chat` |
| 来源记录ID | 文本 | 表单源记录 ID |
| 来源消息ID | 文本 | 群聊源消息 ID |
| 来源组合 | 文本 | 保留表单和群聊来源关联 |
| 日报类型 | 单选/文本 | `单日`、`多日合并` |
| 日期覆盖范围 | 文本 | 多日合并时记录原始范围 |
| 匹配方式 | 单选/文本 | `open_id`、`name_fallback`、`unmatched` |
| 匹配状态 | 单选/文本 | `已匹配`、`姓名匹配`、`未匹配` |
| 合并状态 | 单选/文本 | `单来源`、`重复已合并`、`内容冲突` |
| 冲突状态 | 单选/文本 | `无冲突`、`内容冲突` |
| 事实记录状态 | 单选/文本 | `有效`、`待人工确认`、`忽略` |
| 同步时间 | 日期时间/文本 | 最近一次事实表 upsert 时间 |

事实表只保留每个成员每天一条主记录，原始来源通过 `来源记录ID`、`来源消息ID` 和 `来源组合` 追溯。

## 同步规则

### 群聊日报实时链路

```text
群消息事件
-> 解析日报
-> 写入群聊日报原始表
-> 根据事实唯一键更新日报统一事实表
```

群聊日报解析失败时，仍写入 `群聊日报原始表`，`解析状态=解析失败`，不进入事实表。

### 表单日报定时链路

```text
每日定时任务
-> 读取工作日报表/日报汇总表
-> 规范化字段
-> 根据事实唯一键更新日报统一事实表
```

定时任务也读取 `群聊日报原始表` 做校准，修复实时事件漏处理、服务重启或重复事件导致的不一致。

## 唯一键与匹配规则

事实唯一键：

```text
优先：成员OpenID + 日报日期
兜底：日报提交人姓名 + 日报日期
```

正式组织内优先使用 OpenID，避免重名问题。个人组织或导入数据测试时，如果 OpenID 缺失或失真，使用姓名兜底。

通讯录匹配优先级：

1. 使用 `发送人OpenID` 或表单人员字段解析出的 open_id 匹配通讯录中的当前账号。
2. OpenID 不可用时，用日报标题姓名匹配 `成员真实姓名` 或 `成员别名`。
3. 匹配失败时仍写入事实表，但 `匹配状态=未匹配`，`事实记录状态=待人工确认`。

## 身份映射与账号迁移策略

团队通讯录不能只依赖飞书“人员”字段的显示名。外包或外部账号可能显示为 `用户400276`，这个显示名通常不能在多维表格人员字段中像文本一样手动修改。

团队通讯录建议保留人员字段，同时增加普通文本字段：

| 字段 | 类型建议 | 说明 |
| --- | --- | --- |
| 团队成员 | 人员 | 当前飞书账号 |
| 成员真实姓名 | 文本 | 业务展示名，建议维护 |
| 成员别名 | 文本 | 昵称、简称、日报标题可能出现的名字 |
| 当前OpenID | 文本 | 当前账号 open_id |
| 历史OpenID/历史账号说明 | 文本 | 外包转正式、账号迁移时保留历史 |
| 账号类型 | 单选/文本 | `内部`、`外部`、`个人飞书`、`个人企业`、`正式组织` |
| 成员状态 | 单选/文本 | `在职`、`离职`、`外包`、`已转正式` |
| 所属板块 | 文本 | 用于事实表归属 |
| 敏捷小组 | 文本 | 用于事实表归属 |
| 直属上级 | 人员/文本 | 用于日报管理 |
| 分管领导 | 人员/文本 | 用于后续汇总 |

周报、海报和管理视图优先展示 `成员真实姓名`，不要直接使用人员字段显示名。如果第一期通讯录暂未维护该字段，则按顺序兜底使用日报标题姓名、表单人员字段显示名、飞书人员字段显示名。

外包转正式账号时：

1. 将 `团队成员` 更新为新的正式组织账号。
2. 更新 `当前OpenID`。
3. 保留 `历史OpenID/历史账号说明`。
4. `成员真实姓名` 保持不变。
5. 后续新日报按新账号匹配。
6. 历史事实记录不强行重写，必要时通过真实姓名和历史账号说明关联。

第一期只要求预留这些字段和匹配策略，不做自动账号迁移。

## 多来源合并与冲突规则

内容优先级：

1. 只有表单：事实表使用表单内容。
2. 只有群聊：事实表使用群聊最新主版本内容。
3. 表单和群聊一致：事实表使用表单内容，`日报来源=form+chat`，`合并状态=重复已合并`。
4. 表单和群聊不一致：事实表使用表单内容，`日报来源=form+chat`，`合并状态=内容冲突`，`冲突状态=内容冲突`。

群聊内容不丢弃，始终保留在 `群聊日报原始表`，并通过来源字段关联。

第一版用保守规则判断内容是否一致：

- 去掉首尾空白。
- 统一中文/英文标点差异中的列表编号格式。
- 压缩连续空白和空行。
- 对规范化后的 `今日工作总结 + 明日工作计划 + 遇到的问题` 生成内容指纹。
- 内容指纹相同视为一致；不同视为内容冲突。

第一版不做语义相似度判断，避免 AI 或模糊匹配把不同日报误判为一致。后续可再增加“疑似重复”状态。

## 群聊多版本规则

同一成员同一天在群里多次发送日报：

- 原始表保留每一条群聊消息。
- 最新一条标记为 `主版本`。
- 之前的群聊日报标记为 `历史版本`。
- 事实表使用最新主版本内容。
- 如果当天同时存在表单日报，仍然表单优先。

## 多日合并日报规则

例如：

```text
刘喜双 6.29-6.30 工作日报
```

处理规则：

- 群聊原始表保留一条原始消息。
- 事实表自动拆成多条事实记录：
  - `刘喜双 + 2026-06-29`
  - `刘喜双 + 2026-06-30`
- 每条事实记录都写：
  - `日报类型=多日合并`
  - `日期覆盖范围=2026-06-29~2026-06-30`
- 第一版不强行拆分正文，拆分后的每条事实记录使用同一份工作总结。
- 如果日期范围跨周，拆分后的记录按各自 `日报日期` 进入对应周统计。

## 错误处理

- 群聊消息解析失败：写入 `群聊日报原始表`，不进入事实表。
- 多维表格写入失败：打日志，不影响机器人继续运行。
- 通讯录读取失败：日报照常入原始表，事实表标记未匹配。
- 重复事件：用 `消息ID` 去重。
- 定时任务重复运行：用 `事实唯一键` 和来源 ID upsert，避免重复插入。
- 事实表冲突：不自动覆盖表单主内容，写冲突状态并保留原始来源。

## 配置原则

第一期继续配置驱动：

- 用户手动建表和字段。
- 机器人不自动创建数据表或字段。
- 所有 `appToken`、`tableId`、`viewId` 和字段名都放在 `config/groups.json`。
- 如果事实表或群聊原始表未配置，则相关链路跳过，不影响现有海报和日报解析能力。

## 验收标准

第一期完成后应满足：

- 群里发一条日报，能进入 `群聊日报原始表`。
- 群聊日报能实时生成或更新 `日报统一事实表`。
- 表单日报能通过每日定时任务同步到 `日报统一事实表`。
- 同一人同一天多条群聊日报，事实表以最后一条为准。
- 同一人同一天既有表单又有群聊，事实表以表单为准，并标记是否冲突。
- 多日合并日报会拆成多条事实记录。
- 外部账号显示为 `用户400276` 时，周报和事实表展示仍优先使用 `成员真实姓名`。
- 未匹配人员不会丢数据，会进入待人工确认状态。
- 原有表格/wiki 链接生成海报能力不受影响。
- `npm test` 通过。

## 后续阶段

第一期数据层稳定后，再进入后续阶段：

1. AI 周报摘要草稿，写入待审核表。
2. 分管领导审核和修改。
3. 写入 Wiki 周报模板 Sheet。
4. 生成海报。
5. 定时推送到领导或群聊。
