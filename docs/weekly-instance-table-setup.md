# 周报实例表搭建说明

## 用途

`周报实例表`保存每个 ISO 周对应的 Wiki 工作表实例，是周报复制任务的持久化幂等依据。PM2 重启、手动重跑或定时任务重试时，系统先查询该表；已经存在相同`周报实例唯一键`时，不再复制第二份工作表。

## 字段结构

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| 周报实例唯一键 | 单行文本 | 是 | `ISO年份-W周次`，例如`2026-W29` |
| ISO年份 | 数字 | 是 | ISO 周所属年份 |
| ISO周次 | 数字 | 是 | 取值 1 至 53 |
| 周开始日期 | 日期 | 是 | 本周周一 |
| 周结束日期 | 日期 | 是 | 本周周五 |
| SpreadsheetToken | 单行文本 | 是 | 仅供服务端定位，不发送到群聊 |
| SheetID | 单行文本 | 是 | 复制后的周工作表 ID |
| 工作表名称 | 单行文本 | 是 | 复制后的工作表标题 |
| 周报链接 | 超链接 | 是 | 指向本周工作表 |
| 实例状态 | 单选 | 是 | `已创建`、`创建失败`、`已发布` |
| 创建时间 | 日期时间 | 是 | 首次登记时间，后续更新不覆盖 |
| 更新时间 | 日期时间 | 是 | 最近一次状态更新时间 |

建议把`周报实例唯一键`设为主字段，并创建一个按`周开始日期`降序排列的管理视图。视图可以筛选记录，但服务端查询会绕过视图读取整张数据表，避免筛选条件破坏幂等判断。

## 权限

飞书应用需要对该多维表格具备以下权限：

- 读取记录。
- 新增记录。
- 更新记录。

团队成员可以通过多维表格视图协作查看状态，但不应手工修改`周报实例唯一键`、`SpreadsheetToken`或`SheetID`。如需修复异常实例，应由管理员确认实际 Wiki 工作表后再修改。

## 个人环境配置

表创建后，把 Base/Wiki 链接配置到对应 group 的`weeklyInstanceTable`。日期字段必须声明类型，以便 Node.js 写入毫秒时间戳：

```json
{
  "weeklyInstanceTable": {
    "wikiUrl": "个人组织周报实例表链接，需包含table参数",
    "fieldTypes": {
      "weekStart": "date",
      "weekEnd": "date",
      "sheetUrl": "url",
      "createdAt": "datetime",
      "updatedAt": "datetime"
    }
  }
}
```

首次验收期间保持调度关闭：

```json
{
  "weeklyInstanceCreation": {
    "enabled": false,
    "dayOfWeek": 1,
    "time": "09:00",
    "timezone": "Asia/Shanghai"
  }
}
```

周报工作表轮换是每年一次的人工操作：每年开始使用新的年度模板/工作簿时，由管理员手动确认模板、工作簿和实例表配置；系统不会自动创建或切换年度工作簿。每个工作周仍按幂等键复用已有实例。

工作表命名和位置规则：

- 工作表标题使用该工作周周五的 `MMDD`，默认格式为 `数字金融部周报MMDD`，例如周五为 7 月 17 日时标题为 `数字金融部周报0717`。
- 复制模板后必须把新工作表移动到第一个 tab（索引 `0`）。
- 实例表中的周报链接必须是直接打开复制工作表的链接，并包含复制出的 `sheet` ID；不能只保存工作簿或 Wiki 首页链接。

首次验收期间保持调度关闭。只有完成两次手动执行并确认只产生一份工作表、一个实例记录后，才单独决定是否启用周一自动任务。所有新调度在受控验证完成并取得单独批准前都必须保持禁用。

手动执行命令（仅在已完成受控验证准备、且由操作员明确发起时使用）：

```bash
GROUPS_CONFIG_PATH=config/groups.personal.json npm run weekly:ensure
```

重复执行同一命令应复用同一 ISO 周实例，不得新增第二张工作表或第二条实例记录。该命令不会发送 AI 周报、海报或群消息；交付调度仍需保持关闭。

## 数据安全

- API Key、应用密钥和服务器凭证不得写入该表。
- 验证文档和群聊日志不得记录真实 token、table ID、sheet ID、OpenID 或完整周报链接。
- 普通运行日志只记录组名、ISO 周键、跳过原因和复用状态。
